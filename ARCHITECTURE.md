# Syncd Architecture: The Dual-Database Trigger Pattern

This document outlines how Syncd achieves autonomous two-way synchronization between a local DB (e.g., SQLite) and a remote DB (e.g., PostgreSQL).

## 1. The Core Concept
Instead of application code telling the sync engine what changed, we use Database Triggers. Triggers watch for any `INSERT`, `UPDATE`, or `DELETE`, and automatically write a "Sync Event" (containing the table name, action type, and JSON payload of the row) to an Outbox.

## 2. Externalizing the Outbox 
To address the requirement of keeping the primary database clean:
- **SQLite (Client)**: We can use SQLite's `ATTACH DATABASE` feature. The main database is `app.db`. We attach `syncd_logs.db`. The triggers on `app.db` simply insert records into `syncd_logs.sync_events`.
- **PostgreSQL (Server)**: We can use a completely separate schema (e.g., `CREATE SCHEMA syncd;`), or if physical separation is strictly required, use `postgres_fdw` to write to a sister database. A separate schema is usually highly preferred in PG for performance and simplicity.

## 3. The "Infinite Echo" Loop Problem
**The Problem**: Device A creates a student. The trigger fires. The sync engine pushes that to the VPS. The VPS inserts the student. The VPS's *own* trigger fires, creating a new event. That event syncs *back* to Device A... creating an infinite loop.

**The Solution: Synchronizer Identity Context (The "Ignorer" Pattern)**

For triggers to work in a multi-way sync without looping, the sync engine must identify itself when it writes to the database, so the triggers know to ignore it.

### Postgres Implementation: Session Variables
When the Go Syncd server connects to PostgreSQL to replay a bunch of changes it received from a desktop client, it sets a local session variable first:
```sql
BEGIN;
SET LOCAL syncd.is_replaying = 'true';
-- Do the inserts/updates
COMMIT;
```
The Postgres trigger is written to check this variable:
```sql
CREATE OR REPLACE FUNCTION log_sync_event() RETURNS TRIGGER AS $$
BEGIN
    -- Do not log events that were generated by the syncd engine replaying data
    IF current_setting('syncd.is_replaying', true) = 'true' THEN
        RETURN NEW;
    END IF;
    
    -- Otherwise, insert into the sync_logs table
    -- ...
END;
$$ LANGUAGE plpgsql;
```

### SQLite Implementation: The Flag Table or Temp View
SQLite doesn't have session variables like Postgres. The common workaround is to use a temporary table or a specific column (e.g., `sync_source`).
However, the easiest SQLite approach that doesn't alter your data tables is a `TEMP` table:
When the Go Syncd client connects to replay VPS changes locally:
```sql
BEGIN;
CREATE TEMP TABLE IF NOT EXISTS _syncd_replaying (flag INT);
INSERT INTO _syncd_replaying VALUES (1);
-- Do the inserts/updates
DROP TABLE _syncd_replaying;
COMMIT;
```
And the SQLite trigger checks for its existence:
```sql
CREATE TRIGGER IF NOT EXISTS log_student_insert AFTER INSERT ON students
WHEN NOT EXISTS (SELECT 1 FROM sqlite_temp_master WHERE type='table' AND name='_syncd_replaying')
BEGIN
    INSERT INTO syncd_logs.sync_events (table_name, action, payload) 
    VALUES ('students', 'INSERT', json(NEW));
END;
```

## 4. The Go Daemon Pipeline

The `syncd` binary will run as a persistent daemon. It consists of:
1. **The Watcher**: Periodically polls the attached `syncd_logs.db` (or uses a SQLite WAL listener) to find unsynced outbound events.
2. **The Transporter**: Safely POSTs unsynced local events to the remote server, and GETs new events from the remote server.
3. **The Applicator**: The crucial component that wraps the local SQL connection in a transaction, sets the "Ignorer" flag (`_syncd_replaying`), executes the incoming remote events, and commits.
4. **The Conflict Resolver**: If an inbound event modifies the same record (judged by UUID and `updated_at`) that the local device just modified offline, deterministic rules (typically Last-Write-Wins based on an `updated_at` timestamp inside the payload) dictate if the applicator applies or drops the event.
